# 00 Release Goals

更新时间：2026-03-27

## 1. 发布目标

- 让用户可以稳定完成：`安装 ClawTools -> 配置 Provider/API Key -> 完成诊断 -> 首次可用`
- 将 Windows 作为主发布路径进行治理，Linux/macOS 为兼容路径
- 在修复功能问题的同时，建立一套可以持续复用的 Claude Code 自动开发 + Codex 监督治理模型

## 2. 用户价值目标

- 降低安装失败率
- 降低配置失败率
- 降低“脚本失败但用户看不到原因”的黑盒体验
- 降低由于不安全脚本、执行型配置、SQL 拼接带来的高危风险

## 3. 交付目标

本次不是只修代码，而是同时交付三层能力：

1. 产品可上线
- 安装、配置、诊断主链路稳定可用

2. Claude Code 可自动开发
- agent teams 可基于任务板连续执行，不需要人工逐条盯实现细节

3. Codex 可独立监督
- 不进入主实现回路，但能按固定节奏审查、纠偏、沉淀证据

## 4. 核心用户旅程

1. 用户运行安装入口
2. 用户选择平台/版本并完成安装
3. 用户进入配置页并保存 API Key
4. 用户执行诊断并看到明确结果
5. 用户完成首次 provider 调用

## 5. 本次范围

### In Scope

- 清除全部 P0 发布阻塞
- 建立最小测试与 lint 门禁
- 将 Claude Code 执行路径改为真实可用的自动化 team workflow
- 建立 Codex 的日检、阶段审查、Go/No-Go、复盘责任
- 补齐发布 runbook、验证矩阵、监控与回滚机制

### Out of Scope

- 大规模 UI 重构
- 长周期架构重写
- 与本次首发链路无关的功能扩展
- 让 Codex 成为主开发者

## 6. 成功标准

### 发布成功标准

- 安装成功率 >= 95%
- 配置成功率 >= 95%
- 高危安全事件 = 0
- 无 P0 线上事故

### 执行成功标准

- Claude Code 能基于 `07-agent-team-taskboard.md` 自动持续推进
- Claude 不能无验证关闭任务
- Codex 能独立发现并升级关键偏航
- 文档、任务、验证证据三者一致

## 7. 组织设计原则

- Claude 负责速度和吞吐
- Codex 负责真实性和边界控制
- 所有“完成”都必须能落到证据
- 所有自动化都必须有人工兜底
